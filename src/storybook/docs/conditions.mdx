# Using Conditions

Conditions are named CSS selectors defined in `src/styles/utilities/conditions.ts`. In Panda CSS they become underscore-prefixed props: `_hover`, `_disabled`, `_error`, etc.

**The primary place to use conditions is in recipes.** Recipe `base` and `variants` blocks are statically analyzed at build time, ensuring all styles are included in the output bundle.

```typescript
// src/recipes/button.ts
export const buttonRecipe = defineRecipe({
  base: {
    cursor: 'pointer',
    transitionProperty: 'background, border-color, color, box-shadow',
    transitionDuration: 'fast',
    transitionTimingFunction: 'default',
    _hover:         { bg: 'bg.neutral.hovered' },
    _active:        { bg: 'bg.neutral.pressed' },
    _disabled:      { opacity: '0.3', cursor: 'not-allowed' },
    _focusVisible:  { outlineColor: 'border.focused' },
    _error:         { borderColor: 'border.danger' },
  },
  variants: {
    variant: {
      danger: {
        bg: 'bg.danger',
        _hover: { bg: 'bg.danger.hovered' },
      },
    },
  },
});
```

Conditions can also be applied inline on any JSX element for one-off cases — prototyping, demos, or a single component that doesn't warrant its own recipe:

```tsx
<Box _hover={{ bg: 'bg.neutral.hovered' }}>Hover me</Box>
```

## Data Attribute Conditions

Many conditions respond to both native CSS pseudo-classes **and** data attributes, so you can drive the same visual state from JavaScript.

<Card variant="sunken" p="16">
<VStack alignItems="stretch" gap="8">
<Text weight="bold" color="text">Data attributes must equal the string `"true"`</Text>
<Text>These conditions use strict `[data-x=true]` equality. Pass boolean `true` (React renders it as `"true"`), or use `condition ? true : undefined` so the attribute is omitted rather than set to `"false"`.</Text>

```tsx
// ✅ Correct
<Box data-error={true} />
<Box data-error={hasError ? true : undefined} />

// ❌ Wrong — renders data-error="Field is required", not "true"
<Box data-error={errorMessage} />
```

## Peer Conditions

Peer conditions style an element based on the state of a **preceding sibling** with the `peer` class.
The peer element must appear **before** the target in the DOM (CSS `~` sibling combinator).

## Group Conditions

Group conditions style a **descendant** based on the state of a **parent** with the `group` class.
Several components in this design system use this pattern internally — `Button`, `Chip`, and `FormField` all apply `className="group"` so their child elements can coordinate state.
